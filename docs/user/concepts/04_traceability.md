# 实验可追溯性设计

UESF 的一个核心设计目标是：每次实验结果都能精确追溯到当时使用的配置、代码和数据划分。本文说明这一目标通过哪些机制实现。

---

## 可追溯性的三个维度

| 维度 | 追溯什么 | 存储位置 |
|------|----------|----------|
| 配置 | 实验使用的确切 YAML 参数 | SQLite，以 JSON 快照形式 |
| 代码 | 自定义模型/训练器/指标的确切源码 | SQLite，源码全文快照 |
| 数据 | 哪些样本用于训练/验证/测试 | SQLite，索引快照 |

---

## 配置快照

每次 `uesf experiment run` 时，框架会将实验配置（经过解析后的 Python 对象）序列化为 JSON，存储在数据库的 `experiments` 表中。

这意味着：即使你在实验完成后修改了 `experiments/baseline_cnn.yml`，数据库中仍然保存着**运行时的确切配置**。通过 `experiment query` 看到的结果，始终对应数据库中记录的配置，而不是当前磁盘上的文件。

---

## 源码快照

自定义组件（模型、训练器、指标）在注册或首次被使用时，源文件的**完整文本内容**会被存储在数据库中。

每次 `uesf experiment run` 时，框架检测当前磁盘上的源文件 SHA256 哈希：

- **哈希未变化**：使用数据库中已有记录，不更新
- **哈希变化**：
  1. 将旧记录归档，重命名为 `emotion_cnn_<旧sha256前8位>`，标记为 `is_obsolete=1`
  2. 以原名称 `emotion_cnn` 创建新记录，存储新版本的源码快照
  3. 日志中输出提示：`[INFO] emotion_cnn 源码已更新，旧版本已归档为 emotion_cnn_a1b2c3d4`

这保证任意一次历史实验结果都能找到对应的源码版本，即使原始文件已被多次修改。

---

## seed 与数据划分的可复现性

实验配置中的 `seed` 字段控制数据切分的随机性：

```yaml
seed: 42
```

相同的 `seed`、相同的数据集和相同的切分策略，产生完全相同的 train/val/test 划分。这确保：

1. 同一实验多次运行（例如调试超参数时）使用的是同一份数据划分
2. 对比实验（baseline vs. ablation）在相同的数据划分上进行，对比公平

> 若不指定 `seed`，框架使用随机种子，每次运行的划分可能不同。强烈建议在正式实验中始终指定 `seed`。

## 检查点与实验结果的对应关系

每次实验的检查点保存在：

```
experiments/results/<实验名>/checkpoints/
├── fold_1_best.pt
├── fold_2_best.pt
└── ...
```

数据库中的实验记录包含：

- 实验 ID（唯一标识符）
- 运行时间戳
- 配置 JSON 快照
- 指标结果 JSON
- 组件版本（模型/训练器/指标的 SHA256 记录）

通过实验名或 ID，你可以精确找到：使用了哪个版本的模型代码、哪些数据参与了训练、最优检查点在哪里。

---

## 查看组件历史版本

```bash
# 显示所有版本，包括已归档的旧版本
uesf model list --show-obsolete
```

旧版本格式：`emotion_cnn_a1b2c3d4`（原名称 + SHA256 前 8 位）。数据库中保存了对应的完整源码，可以重建任何历史版本的组件。
